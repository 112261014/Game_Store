一、 系統架構與環境原則
三大獨立 Codebase：

Server：部署於系計 Linux 工作站，需包含 DB/Data（可直接用sqlite 檔案）、Developer Server、Lobby Server。

Developer Client：開發者用，負責開發、測試、上架。

Player Client：玩家用，負責瀏覽、下載、遊玩。

資料持久性 (Persistence)：Server 重啟後，使用者資料與上架遊戲資訊不得遺失。

檔案隔離與版本管理：

Server 端：需有統一的「上架遊戲存放區」，作為下載來源。

Player 端：每位玩家需有「獨立的下載資料夾」（模擬多人在同一台電腦 Demo），玩家不可修改下載內容。

版本一致性：需處理 Developer 本地版、Server 上架版、Player 下載版三者間的同步。

統一遊戲規格：制定標準的檔案結構與 Config 檔，讓平台能統一啟動不同開發者寫的遊戲（CLI/GUI）。

二、 介面與 UX 設計原則 (Menu-driven)
純選單互動：全程使用選單（輸入 1-5 選項），禁止依賴隱藏指令或複雜參數。

啟動限制：僅允許在程式「最初啟動時」使用 Shell 指令或參數，進入程式後需全選單操作。

層級扁平化：每一層選單功能建議不超過 5 個，避免資訊過載。

流程連貫性：避免碎片化操作（如：不要讓使用者手抄 ID 再去別處輸入），應支援「選擇遊戲 → 自動建立房間」的一條龍流程。

非阻塞 UI：輸入介面不應因背景事件（如其他人加入房間）而卡死或畫面錯亂。

三、 帳號與權限原則
身分隔離：Developer 帳號與 Player 帳號完全獨立，資料庫不共用。

唯一登入 (Single Session)：同一帳號同一時間只能有一個有效登入（踢掉舊的或拒絕新的）。

權限控管：

開發者只能管理（更新/下架）自己上傳的遊戲。

玩家只能評分自己實際玩過的遊戲。

四、 核心功能需求
Developer 端：

上架：需包含遊戲檔案 + 設定檔（Config）。例如：
    - 遊戲名稱
    - 簡短介紹或描述
    - 遊戲類型（CLI / GUI / 人數上限等）
    - 要上傳的遊戲位置（或由系統提供可選的清單）
    - 遊戲相關的設定檔（例如 config 檔），可以選擇手動輸入設定，也可以直接由配置檔讀取，確保上架內容本身就包含啟動與連線所需的資訊

更新：版本更新後，Server 需標記為最新，舊版玩家應被引導更新。

下架：下架後不可再被下載或開新房間。

Player 端：

瀏覽：顯示列表與詳細資訊（簡介、評分）。

下載/更新：遊玩前自動檢查並下載最新版。

遊玩：自動啟動 Game Client 並連線至 Game Server，結束後釋放資源回大廳。

評分：結束遊戲後可評分與留言。

五、 必抓的例外狀況 (Exception Handling)
這部分是 Demo 時助教會攻擊的重點，請務必處理：

1. 網路與連線異常
Server 斷線/連線逾時：在上架、下載、瀏覽列表時若 Server 無回應，需提示錯誤並提供重試/取消選項，不可直接 Crash。

下載失敗/檔案損毀：下載中斷時，應標記失敗，不可將半殘檔案視為成功，需支援重新下載。

2. 輸入與資料驗證
無效輸入：選單輸入非數字、超出範圍（如 1-5 輸入 9）、評分超出範圍（0 或 100 分）。

惡意/髒資料：空白名稱、重複的帳號名稱、過長的評論字串。

缺少必要欄位：上架時 Config 檔缺少啟動指令或 Port。

3. 邏輯與狀態衝突
版本衝突：玩家持有舊版但 Server 已有新版（需提示更新）。

資源不存在：

欲下載的遊戲已被下架。

欲加入的房間已滿或遊戲已結束。

開發者試圖更新不屬於自己的遊戲。

未玩先評：嘗試對未執行過的遊戲進行評分。

4. 檔案系統錯誤
路徑錯誤：上架時選取的本地路徑不存在。

權限錯誤：無法寫入下載資料夾。

六、 Demo 準備建議 (DevOps)
自動化腳本：提供 Makefile 或 python venv 等方式，讓助教能一鍵安裝環境並啟動 Server/Client。

README：清楚條列啟動步驟，把助教當成「完全不懂你 Code 的人」。

測試資料：準備腳本以便在 Demo 前快速清空 DB 並注入一組預設的測試資料（如幾個已上架的遊戲）。
